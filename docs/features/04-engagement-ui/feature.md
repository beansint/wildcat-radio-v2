# Feature: Engagement UI + studio console  (Issue #7)

- **Branch:** `feature/7-engagement-ui`  ·  **Milestone:** M3  ·  **Repo:** frontend
- **Spec:** backend `docs/final-build-plan/00-MASTER-SPEC.md` section 5.4, `docs/features/010-engagement/`

## Problem & why

Slice 3 gives listeners a live room without letting participation interrupt the broadcast. The UI needs to keep queue submissions private until the booth acts, show polls and hype as aggregate state, and give the station PC a fast token-gated console for queue, poll, pin, and booth-chat operations.

## Acceptance criteria

- **AC-1:** `/listen` renders chat, poll, fixed reactions, hype meter, pinned topic, up-next, and request/dedication/Q&A entry points without blocking stream playback.
- **AC-2:** Anonymous and unverified listeners cannot send queue/chat/poll writes, see the correct CTA, and can still see the listening room.
- **AC-3:** A verified listener submits request, dedication, or Q&A to `POST /api/episodes/:id/queue`; the sheet clears, closes, and shows a private pending receipt state.
- **AC-4:** Listener chat sends through `chat:message`; `chat:new` appends messages in feed order and booth messages render as the booth identity.
- **AC-5:** Polls load from `GET /api/episodes/:id/polls`, voting uses `POST /api/polls/:id/vote`, `poll:updated` refreshes counts, and duplicate vote errors remain non-destructive.
- **AC-6:** Fixed reactions use only `🔥`, `❤️`, `😂`, and `👏`; `hype:tick` updates the aggregate hype meter.
- **AC-7:** `queue:receipt`, `queue:up-next`, and `topic:pinned` update private receipts, public up-next, and pinned topic surfaces.
- **AC-8:** `/studio` unlocks with a pasted station token, stores it locally, and sends it as `Authorization: Bearer <token>` to generated studio endpoints.
- **AC-9:** `/studio` lists the active queue and can queue, read, or silently decline items with clear status feedback.
- **AC-10:** `/studio` can launch and close polls, set a pinned topic, post booth chat, and display hype updates.
- **AC-11 (edge):** No live episode, invalid token, over-budget queue submit, duplicate poll vote, and silent decline all show the expected UI state without leaking declined listener content.

## Scope

- **In:** `/listen`, `/studio`, generated Orval engagement/studio clients, Socket.IO listener room events, feature docs, and Playwright coverage.
- **Out:** moderation queue, content filtering, staff roster CRUD, attendance time-in UI, and production deployment.

## Contract

- **REST:** `POST /api/episodes/:id/queue`, `GET /api/episodes/:id/polls`, `POST /api/polls/:id/vote`, `POST /api/episodes/:id/reactions`, `GET /api/studio/queue`, `POST /api/studio/queue/:id/act`, `POST /api/studio/polls`, `POST /api/studio/polls/:id/close`, `POST /api/studio/pin`, `POST /api/studio/chat`.
- **Socket.IO:** client emits `episode:join`, `episode:leave`, `chat:message`; server emits `chat:new`, `chat:hidden`, `queue:receipt`, `queue:up-next`, `poll:updated`, `hype:tick`, `topic:pinned`.
- **Generated files:** frontend `openapi/openapi.json`, `src/lib/api/endpoints/engagement/engagement.ts`, `src/lib/api/endpoints/studio/studio.ts`, and `src/lib/api/model/*`.

## Edge cases

- Guest third queue item in one episode returns the backend budget error and no optimistic duplicate.
- `DECLINE` returns success to studio but produces no listener receipt and no up-next item.
- Duplicate poll vote keeps the existing selected option stable and shows a non-blocking error.
- Invalid station token leaves `/studio` locked with a clear error.
- Socket reconnect rejoins the current episode room and resumes live event handling.

## Notes / decisions

- POST operations are generated by Orval as query hooks, so runtime UI calls the generated async functions through local TanStack Query mutations.
- `/studio` is a station-token surface, not Better Auth user login. Attendance time-in UI remains FE #5.
- Forms use shadcn/ui primitives with `react-hook-form` and `zod`; each active form exposes one `role="alert"` region.
